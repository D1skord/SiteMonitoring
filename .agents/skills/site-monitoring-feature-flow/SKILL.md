# Site Monitoring Feature Flow

Используй этот skill для обычной разработки в Site Monitoring: фичи, багфиксы, рефакторинг, тесты и изменения PHP/Symfony-кода.

## Цель

Довести изменение до проверяемого результата с минимальным diff:

1. понять задачу и критерий успеха;
2. найти ближайший существующий код и тесты;
3. сделать точечную правку;
4. проверить результат объективной командой;
5. кратко сообщить, что изменено и что проверено.

## Правила

- Общайся с пользователем на русском, если он не попросил иначе.
- Перед правками проверь `git status --short --branch`.
- Не трогай чужие незакоммиченные изменения.
- Не читай и не меняй `docker/data/`.
- Не раскрывай значения secrets из `.env*`, Docker compose и CI.
- Для поиска используй `rg`/`rg --files`, исключая `vendor/`, `var/`, `docker/data/`, `.git/`, `.idea/`.
- Меняй только файлы, напрямую связанные с задачей.
- Не добавляй абстракции, конфигурацию или универсальность без явного запроса.

## Flow

1. Сформулируй допущения и критерий успеха.
2. Найди релевантные файлы:
   - бизнес-логика: `src/Service`, `src/Command`, `src/MessageHandler`;
   - HTTP/Twig/Form: `src/Controller`, `src/Form`, `templates`;
   - Doctrine: `src/Entity`, `src/Repository`, `migrations`;
   - тесты: `tests/Unit`, `tests/Functional`, `tests/Integration`.
3. Изучи существующий стиль рядом с изменением.
4. Если поведение можно зафиксировать тестом, добавь или обнови ближайший релевантный тест.
5. Внеси минимальную правку.
6. Удали только тот мертвый код, который появился из-за этой правки.
7. Проверь результат:
   - для измененных PHP-файлов минимум `php -l path/to/file.php`;
   - для бизнес-логики релевантный PHPUnit-тест или `make phpunit`;
   - для широких изменений `make test`;
   - если проверка невозможна из-за окружения, прямо скажи, что не выполнено и почему.

## Локальные ориентиры

- Для новой бизнес-логики сначала ищи существующий сервис, команду или тест рядом с нужным сценарием.
- Для CRUD и форм держись Symfony patterns из `SiteController`, `SiteType`, Twig-шаблонов.
- Для новых уведомителей реализуй `NotifierHandlerInterface`.
- Для фоновых сценариев используй Symfony Command и при необходимости Messenger.
- Для изменений схемы создавай Doctrine migration, не редактируй БД вручную.
- Для проверки дат, HTTP и уведомлений добавляй unit-тесты на чистую логику; functional/integration-тесты используй только если нужна БД или контейнер Symfony.

## Отчет пользователю

В финальном ответе кратко укажи:

- что изменено;
- какие проверки запущены;
- какие проверки не удалось выполнить, если такие есть;
- есть ли остаточные риски или follow-up.
